爆速静的サイトジェネレータHUGO

June 9, 2019

静的サイトジェネレータとしてJekyllを使っていたが、ビルドがとにかく遅い。
本当に遅い。
そんなところでストレスためてもよくないので環境を引っ越すことにした。
以下をつらつらと眺めて、速度が売りのHUGOに決めた。
https://www.staticgen.com/

実際に使ってみたが、サイトのビルドは底抜けに速い。
本当にできあがったのか、心配になるくらい。でもできてる。
速度に関しては今のところ、大変に満足。

爆速サイトジェネレータHUGO

https://gohugo.io/
HUGOとは、GO言語で書かれた「世界一速いウェブサイト構築フレームワーク」である。
「ウェブサイト構築をもう一回、楽しくする！」

The world’s fastest framework for building websites.
Hugo is one of the most popular open-source static site generators. With its amazing speed and flexibility, Hugo makes building websites fun again.

威勢がよろしい。

HUGOを使ったサイト作成の流れ

典型的な流れを示す。

  1. hugo用の作業ディレクトリを作る
  2. テーマをインストールする
  3. 設定ファイルを編集する
  4. おもむろに記事を書く。この時点ではドラフト版。
  5. ビルドしてローカルホスト上で見た目を確認する
  6. 記事を正式版にする。
  7. ビルド、デプロイ

作業ディレクトリの作成や、ローカルホストでの見た目確認、ビルドはhugoコマンドを使っていくことになる。
また、初回は1から7を通して実施するが、構築が済んでしまえば、4～7を繰り返す。

HUGO作業用ディレクトリを作る

ここを見ながら。
https://gohugo.io/getting-started/quick-start/

まずバージョンを確認しよう。

# hugo version
Hugo Static Site Generator v0.54.0 freebsd/amd64 BuildDate: unknown

作業用ディレクトリを作る。
書式はhugo new site <ディレクトリ>

# hugo new site lair

Just a few more steps and you're ready to go:

   with "hugo new <SECTIONNAME>/<FILENAME>.<FORMAT>".

出来上がったディレクトリ構成は以下の通り。

# tree lair
/lair
|-- archetypes
|   `-- default.md
|-- config.toml
|-- content
|-- data
|-- layouts
|-- static
`-- themes

config.tomlが設定ファイル。
記事はcontentに入れる。
画像ファイルなどはstaticに入れる。

static file https://gohugo.io/content-management/static-files/

テーマをインストール

まずはガイドの通りにanankeをインストールする。
インストールといっても、gitでダウンロードしthemes/anankeに格納するだけ。
config.tomlにanankeをthemeとして使うよう書き込む。

# git init
Initialized empty Git repository in /usr/home/lair/lair/.git/


# git submodule add https://github.com/budparr/gohugo-theme-ananke.git themes/ananke
Cloning into '/usr/home/lair/lair/themes/ananke'...
remote: Enumerating objects: 1, done.
remote: Counting objects: 100% (1/1), done.
remote: Total 1333 (delta 0), reused 0 (delta 0), pack-reused 1332
Receiving objects: 100% (1333/1333), 4.13 MiB | 447.00 KiB/s, done.
Resolving deltas: 100% (719/719), done.

# echo 'theme = "ananke"' >> ./config.toml

設定ファイルの編集

config.tomlをさらに編集。
これくらいやっておけば。

baseURL = "https://nobwak.github.io/"
languageCode = "ja-JP"
title = "Nobwak's Lair"
theme = "ananke"
# defaultContentLanguage = "ja"
googleAnalytics = ""
hasCJKLanguage = true

configure HUGO https://gohugo.io/getting-started/configuration/

defaultContentLanguageはjaでよさそうだが、リストページ（複数の記事へのリンクをまとめたページ）にある「read more」ボタン等が表示されない。
何かしら設定があるのだとは思うが、深追いせず、いったんコメントアウトしている。

googleAnalyticsは、googleAnalyticsのTracking IDを入れる。するとthemeがそれを読んでくれる。
ただこの件については稿を改めて記す。

hasCJKLanguageはtrueにしておくこと。
こうしないとリストページに表示される各記事のサマリがうまく表示されない。（みっしり表示される）
各記事のFront Matterでも操作できるが、config.tomlで設定するのはグローバルオプション。

もう少し詳しい説明はこちらを参照。
https://kodify.net/hugo/config/configuration-variables/

ためしに記事を書く

書式はhugo new <ファイル>
hugo用のfront matter（設定内容）の書かれたmarkdownを作ってくれる。
もともとあるmarkdownを使ってもよいし、他のエディタでファイルを作ってもよいが、その場合front matterは自分で書くこと。
わたくしの場合、記事はVisualStudioCodeで書いているのでsnippetを作った。

# hugo new posts/my-first-post.md
/home/lair/lair/content/posts/my-first-post.md created
#

できたファイルの内容は以下の通り。

---
title: "My First Post"
date: 2019-06-07T18:00:59+09:00
draft: true
---

最低限必要なfront matterが書かれている。

ここでdraft: trueとあるように、この記事がドラフト版であると示されている。
これは、書きかけの記事が意図せず公開されてしまうことを防ぐため。
逆に言うと、正式公開してよいなら、ここをfalseに変えてビルド、デプロイすればよい。

お試しビルド、見た目確認

書式はhugo server -D。
serverとは、ローカルホストで確認するためにビルドしwebサーバを動かすということ。
-Dはドラフト版の記事を見せてね、という意味。

注意すべきは、この段階でのビルドは、あくまでもローカルホストで確認するためのもの、ということ。
本番ビルドは引数なしのhugoで実行される。

記事を正式版に

問題なければ、記事のfront matterでdraftをfalseとし、hugo serverと今度は-Dオプションなしで実行。
もう一回確認。飛ばしてもいいけど。

# cat ./content/posts/my-first-post.md
---
title: "My First Post"
date: 2019-06-07T18:00:59+09:00
draft: false
---

ビルド、デプロイ

さらに今度はhugoと一切のオプションなしで実行。

# hugo

                   | JA
+------------------+----+

  Paginator pages  |  0
  Non-page files   |  0
  Static files     |  3
  Processed images |  0
  Aliases          |  1
  Sitemaps         |  1
  Cleaned          |  0

Total in 16 ms

オプションなしのhugoを実行すると、publicディレクトリにすべてのファイルが配置される。
「すべての」というのは、content/postsの下にある記事だけでなく、staticに置いたファイル（例えば画像とか）も全部まとめて、ということである。
つまり、本番サーバにはここのpublicにあるファイルをデプロイすればよろしい。
ただし。themeにanankeを使っている場合には、ちょっとした手順が必要だ。
それについては稿を改めて記す。

# tree -L 2 .
.
|-- archetypes
|   `-- default.md
|-- config.toml
|-- content
|   `-- posts
|-- data
|-- deploy.sh
|-- layouts
|-- public
|   |-- 404.html
|   |-- categories
|   |-- dist
|   |-- images
|   |-- index.html
|   |-- index.xml
|   |-- posts
|   |-- sitemap.xml
|   |-- tags
|   `-- wp-content
|-- resources
|   `-- _gen
|-- static
|   `-- wp-content # ここのファイルがpublicの下にもある。
`-- themes
    `-- ananke
